# 一键启动（推荐）

在仓库根目录执行以下脚本，会自动完成：安装依赖 → 编译主项目 → 编译 Control UI → 启动 Gateway → 打开 Dashboard。Dashboard 默认语言为**中文**。

```bash
./start.sh
```

按 Enter 可关闭 Gateway 并退出；或直接关闭终端会一并结束进程。

- **同一 session**：Gateway 与 Dashboard 均使用 dev 端口 **19001**，避免两个进程抢端口；脚本会等 Gateway 就绪后再打开 Dashboard。
- **Ollama**：若本机已运行 `ollama serve`（默认 127.0.0.1:11434），可设 `OLLAMA_HOST=127.0.0.1:11434`（或 `http://127.0.0.1:11434`）以便正确发现模型；未设置时默认使用该地址。

# 先准备好环境（手动执行时）

```bash
pnpm install 或者 pnpm install --no-optional
pnpm build
pnpm ui:build   # 需要用到 Control UI 时再执行
```

# 启动本地 OpenClaw（手动分步）

**必须在仓库根目录执行**（保证 `node_modules` 和 `dist/` 同级，否则会报找不到 chalk 等依赖）。

1. **启动 Gateway**（保持终端不关）

   ```bash
   cd /path/to/openclaw
   OPENCLAW_SKIP_BUILD=1 OPENCLAW_SKIP_CHANNELS=1 pnpm gateway:dev
   ```

   若你本地已有完整构建且不想触发自动构建，可加 `OPENCLAW_SKIP_BUILD=1`（避免因未提交更改触发失败的重建）。

2. **打开带 token 的 Control UI**（新开终端，同样在仓库根目录）
   ```bash
   cd /path/to/openclaw
   pnpm openclaw --dev dashboard
   ```
   用浏览器打开输出的链接（如 http://127.0.0.1:19001/#token=...）。Control UI 在未保存过语言偏好时**默认显示中文**。

# 开启任务：让 OpenClaw 开始工作

本地 Gateway 和 Dashboard 跑起来后，可以用下面任一方式**下发任务**，OpenClaw 会通过 Gateway 调用模型并执行（写代码、查资料、总结等）。

## 方式一：Control UI（浏览器，推荐）

1. 用浏览器打开 `pnpm openclaw --dev dashboard` 输出的链接（带 `#token=...`）。
2. 若提示未连接，在设置里确认 **Gateway 地址**（dev 下为 `ws://127.0.0.1:19001`）并粘贴 **Token**，点击连接。
3. 在左侧选好**会话**（默认 `main` 即可），在底部输入框里输入你的任务，例如：
   - 「总结一下这份文档的要点」
   - 「帮我写一个 Python 脚本，读取当前目录下的 CSV 并输出前 5 行」
   - 「用 union-search 查一下 GitHub 上 openclaw 相关仓库」
4. 回车发送。Agent 会开始推理、可能调用工具，回复会流式显示在聊天里；需要中途停止可点 **Stop** 或输入 `/stop`。

适合：日常对话式任务、多轮协作、看会话历史和工具调用。

## 方式二：命令行一行任务

在新终端、仓库根目录执行（确保本机 Gateway 在跑）：

```bash
cd /path/to/openclaw
pnpm openclaw agent --message "你的任务描述"
```

例如：

```bash
pnpm openclaw agent --message "列出当前目录下的所有 .md 文件并简要说明每个的用途"
```

回复会直接打印在终端；dev 下默认走 Gateway（`http://127.0.0.1:19001`）。要指定会话或把回复发到某个渠道，可加 `--session-id`、`--deliver --channel telegram` 等，详见 `openclaw agent --help`。

适合：单次任务、脚本/自动化、不依赖浏览器。

## 方式三：TUI（终端内聊天）

```bash
pnpm openclaw tui
```

会连接本机 Gateway，在终端里出现聊天界面；输入消息回车即可下发任务，和 Control UI 类似，只是界面在终端。适合无图形环境或习惯终端的场景。

## 配置文件位置（在这里改）

- **dev 模式**（`pnpm gateway:dev` / `./start.sh`）：**`~/.openclaw-dev/openclaw.json`**

## 为什么 `ollama list` 里的模型在 OpenClaw 里“没识别出来”？（含 Ollama 常驻仍报 fetch failed）

OpenClaw 在 **Gateway 启动时** 才会去发现 Ollama 模型（用 Node 的 `fetch` 请求本机 `http://127.0.0.1:11434/api/tags`）。**发现只做一次，不会在运行中自动重试。**

即使你本机 Ollama 常驻、浏览器或 `curl http://127.0.0.1:11434/api/tags` 都能访问，Gateway 进程里仍可能报 **`Failed to discover Ollama models: TypeError: fetch failed`**，常见原因：

- **代理环境**：若系统或终端里设了 `HTTP_PROXY` / `HTTPS_PROXY`，Node 的 fetch 可能把 127.0.0.1 也走代理，导致连不上。解决：启动 Gateway 前设 `NO_PROXY=127.0.0.1,localhost` 或 `no_proxy=127.0.0.1,localhost`，或临时 unset 代理再起 OpenClaw。
- **发现超时**：启动时 Ollama momentarily 无响应会直接失败。可改用下面“做法二”**不依赖发现**，在配置里显式写死 Ollama 的 `baseUrl` 和 `models`，就不会再发请求发现。

可选两种做法：

### 做法一：让“自动发现”成功（先起 Ollama，再起 OpenClaw）

1. 先在本机启动 Ollama：`ollama serve`（或已常驻则不用管）。
2. 再启动 OpenClaw（`./start.sh` 或 `pnpm gateway:dev`）。  
   `start.sh` 里已设置 `OLLAMA_API_KEY=ollama-local`；若你手动起 gateway，请同样加上：  
   `export OLLAMA_API_KEY=ollama-local` 再执行 `pnpm gateway:dev`。
3. 若 Ollama 不在默认地址，可设 `OLLAMA_HOST=127.0.0.1:11434`（或 `http://127.0.0.1:11434`）再起 OpenClaw。

这样启动时发现会成功，`ollama list` 里能看到的模型会出现在 OpenClaw 的模型列表里。若已经先起了 OpenClaw 再起的 Ollama，需要**重启一次 Gateway** 才会重新发现。

### 做法二：不依赖发现，在配置里显式写出模型（推荐，稳定）

不依赖自动发现，直接在 **`~/.openclaw-dev/openclaw.json`** 里写上 Ollama 的地址和模型列表，OpenClaw 会一直识别这些模型。

        "models": [
          {

            "name": "qwen3.5",
            "reasoning": false,

            "contextWindow": 32768,
            "maxTokens": 8192
          },
          {
            "id": "deepseek-r1:1.5b",
            "name": "deepseek-r1:1.5b",
            "reasoning": true,
            "input": ["text"],
            "cost": { "input": 0, "output": 0, "cacheRead": 0, "cacheWrite": 0 },
            "contextWindow": 32768,
            "maxTokens": 8192
          },
          {
            "id": "gpt-oss:latest",
            "name": "gpt-oss",
            "reasoning": false,
            "input": ["text"],
            "cost": { "input": 0, "output": 0, "cacheRead": 0, "cacheWrite": 0 },
            "contextWindow": 32768,
            "maxTokens": 8192
          },
          {
            "id": "phi4-reasoning:latest",
            "name": "phi4-reasoning",
            "reasoning": true,

            "cost": { "input": 0, "output": 0, "cacheRead": 0, "cacheWrite": 0 },

把你本机 `ollama list` 里要用的模型按上面格式补进 `models` 数组即可；`id` 用 `NAME` 那一列（如 `qwen3.5:latest`）。保存后无需重启，配置会热加载；若 Gateway 已跑着，下次发消息或打开模型选择即可看到。

## 选择并设置默认使用的模型

**格式**：`provider/model`，Ollama 即 `ollama/模型名`。

1. **只改默认主模型**（推荐先做）：

   ```bash
   # dev 下改第一个 agent 的默认模型（常用，模型名按你本机 ollama list 的 NAME）
   pnpm openclaw --dev config set agents.list.0.model.primary "ollama/qwen3.5:latest"
   ```

   或改全局默认（对所有未单独指定模型的 agent 生效）：

   ```bash
   pnpm openclaw --dev config set agents.defaults.model.primary "ollama/qwen3.5:latest"
   ```

2. **Ollama 下建议用支持工具调用的模型**（避免 `does not support tools`）：  
   你本机已有的如 `qwen3.5:latest`、`gpt-oss:latest`、`deepseek-r1:1.5b`、`phi4-reasoning:latest` 等，在配置里写进 `models.providers.ollama.models` 或让自动发现成功后，即可设为默认模型。

3. **主模型 + 回退**（可选）：

   ```bash
   pnpm openclaw --dev config set agents.defaults.model '{"primary":"ollama/qwen3.5:latest","fallbacks":["ollama/gpt-oss:latest"]}'
   ```

修改后**无需重启** Gateway，下次发消息会按新配置选模型（若已连上 Control UI，直接再发一条即可）。

## 调用方式与 LLM 参数

- **调用方式**：Gateway 收到聊天请求后，用配置里的 `agents.defaults.model`（或对应 `agents.list[].model`）解析出 `provider/model`，再根据 provider 调对应 API（Ollama 即本机 `http://127.0.0.1:11434` 原生 `/api/chat`，支持流式 + 工具调用）。
- **常用 LLM 相关配置**（均在 `~/.openclaw-dev/openclaw.json` 或非 dev 的 `~/.openclaw/openclaw.json` 里）：

| 配置项                                        | 含义                                      | 示例                                        |
| --------------------------------------------- | ----------------------------------------- | ------------------------------------------- |
| `agents.defaults.model.primary`               | 默认主模型                                | `"ollama/qwen2.5-coder:32b"`                |
| `agents.defaults.model.fallbacks`             | 主模型不可用时的回退列表                  | `["ollama/llama3.3"]`                       |
| `agents.defaults.models["ollama/xxx"].params` | 某模型的参数（temperature、maxTokens 等） | `{ "temperature": 0.7, "maxTokens": 8192 }` |
| `agents.defaults.contextTokens`               | 上下文长度上限（约）                      | `200000`                                    |
| `agents.defaults.thinkingDefault`             | 默认思考档位                              | `"low"` / `"off"`                           |
| `agents.defaults.timeoutSeconds`              | 单次 run 超时（秒）                       | `600`                                       |

按模型设置参数示例（CLI）：

```bash
# 为 ollama/qwen2.5-coder:32b 设置 temperature、maxTokens
pnpm openclaw --dev config set 'agents.defaults.models["ollama/qwen2.5-coder:32b"].params' '{"temperature":0.7,"maxTokens":8192}'
```

也可在 **Control UI → Config** 里用 Raw JSON 编辑同一文件。保存后配置会热加载。

## 外部推理服务（OpenAI、Claude、Gemini、GLM、MiniMax、Kimi 等）在哪里配置？有界面吗？

**配置位置**：和上面一样，都是 **`~/.openclaw-dev/openclaw.json`**（dev）或 `~/.openclaw/openclaw.json`（非 dev）。  
**有界面**：Control UI 里 **Config** 支持两种方式：

1. **表单模式**：按配置 schema 生成表单，可填 baseUrl、apiKey、模型等（若有对应 schema）。
2. **Raw JSON 模式**：直接编辑整份 JSON，适合 `models.providers`、`env`、`agents.defaults.model` 等。

入口：打开 Dashboard → 左侧 **Config** → 可切换 **Form** / **Raw**，改完点保存（Apply），配置会热加载。

### 配置方式概览

| 类型                               | 说明                                                                                                                                                                                   |
| ---------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **内置 provider（只配 key 即可）** | OpenAI、Anthropic(Claude)、Google(Gemini)、Z.AI(GLM)、MiniMax、Kimi 等多数通过 **环境变量** 或 **auth profile** 配 API key，无需在 config 里写 baseUrl（除非用自建/代理）。            |
| **环境变量**                       | 例如 `OPENAI_API_KEY`、`ANTHROPIC_API_KEY`、`GEMINI_API_KEY`、`ZAI_API_KEY`、`MINIMAX_API_KEY`、`MOONSHOT_API_KEY`（Kimi）。可在 config 的 `env` 里写，或导出到启动 Gateway 的 shell。 |
| **自定义 baseUrl / 模型**          | 在 **`models.providers`** 里为某 provider 写 `baseUrl`、`apiKey`、`models` 等，例如自建 OpenAI 兼容、或指定 Kimi/MiniMax 的 API 地址。                                                 |

### 示例：仅用 API key（推荐先试）

在 config 的 `env` 里写上 key（或 export 后起 Gateway），再设默认模型即可：

```json
{
  "env": {
    "OPENAI_API_KEY": "sk-...",
    "ANTHROPIC_API_KEY": "sk-ant-...",
    "GEMINI_API_KEY": "...",
    "ZAI_API_KEY": "...",
    "MOONSHOT_API_KEY": "..."
  },
  "agents": {
    "defaults": {
      "model": { "primary": "anthropic/claude-sonnet-4-5" }
    }
  }
}
```

内置 provider 的默认 baseUrl 见官方文档，一般不用改。

### 示例：自定义 baseUrl + 模型（Kimi / MiniMax / 自建等）

要用自建或指定某服务的 baseUrl、模型列表时，在 **`models.providers`** 里加一段，例如 Kimi（Moonshot）：

```json
{
  "models": {
    "providers": {
      "moonshot": {
        "baseUrl": "https://api.moonshot.ai/v1",
        "apiKey": "sk-...",
        "api": "openai-completions",
        "models": [{ "id": "kimi-k2.5", "name": "Kimi K2.5" }]
      }
    }
  },
  "agents": { "defaults": { "model": { "primary": "moonshot/kimi-k2.5" } } }
}
```

MiniMax、GLM、OpenAI 自建等同理：在 `models.providers` 下新建一个 key，写 `baseUrl`、`apiKey`、`api`（如 `openai-completions` / `anthropic-messages`）、`models`。  
**模型参数**（如 temperature、maxTokens）写在 `agents.defaults.models["provider/model"].params` 里，或在 Control UI Config 表单里对应字段填写。

更细的 provider 列表与字段说明见官方文档：[Model Providers](https://docs.openclaw.ai/concepts/model-providers)、[Gateway 配置参考](https://docs.openclaw.ai/gateway/configuration-reference)。

# 常见报错与处理

## 1. `pnpm build` 卡在 A2UI / `Error: missing dist/entry.(m)js`

**原因**：`pnpm build` 先执行 `pnpm canvas:a2ui:bundle`，这一步失败会导致后面的 `tsdown` 不执行，`dist/entry.js` 没有生成；而 `pnpm gateway:dev` 会检测 dist 过期并自动跑构建，若此时 pnpm 报 “No projects matched the filters” 或构建失败，就会出现 “missing dist/entry” 。

**做法一（推荐）**：只生成 CLI 入口，跳过 A2UI bundle（不搞 Canvas 时可这样先跑通）。

若提示 **`Command "tsdown" not found`**：说明依赖没装全（例如之前 `pnpm install` 因 404 中断）。**先**在仓库根目录执行：

```bash
pnpm install --no-optional
```

装好后再跑 tsdown，任选其一：

```bash
pnpm exec tsdown --no-clean
# 或
node node_modules/tsdown/dist/run.mjs --no-clean
```

然后执行后处理脚本：

```bash
node --import tsx scripts/write-build-info.ts
node --import tsx scripts/write-cli-startup-metadata.ts
node --import tsx scripts/write-cli-compat.ts
```

再执行：

```bash
pnpm gateway:dev
```

**做法二**：修好 A2UI 再完整构建。确保已安装依赖后执行：

```bash
pnpm canvas:a2ui:bundle
```

若报错，看终端里是 `tsc` 还是 `rolldown` 失败，缺依赖就 `pnpm install`，再重试。成功后再：

```bash
pnpm build
pnpm gateway:dev
```

## 2. `pnpm install` 报 404：`whiskeysockets/libsignal-node` Not Found

**原因**：Baileys（WhatsApp）依赖的 `libsignal-node` 从 GitHub codeload 拉取特定 commit 的 tarball，该地址有时返回 404（上游或网络问题）。

**说明**：`pnpm install --no-optional` **不能**绕过此错误，因为 libsignal 是 baileys 的必需依赖，不是 optional。

**做法**（任选其一）：

- **重试 / 换网络**：过一段时间或换网络/代理后再执行 `pnpm install`。
- **用本地 clone 绕过 404**：若 404 持续，可本地打包后让 pnpm 用本地包：
  1. 克隆并打包：`git clone https://github.com/WhiskeySockets/libsignal-node.git /tmp/libsignal-node && cd /tmp/libsignal-node && git checkout 1c30d7d7e76a3b0aa120b04dc6a26f5a12dccf67 && npm pack`
  2. 在仓库根目录的 `package.json` 的 `pnpm.overrides` 里增加一行：`"libsignal": "file:/tmp/libsignal-node/whiskeysockets-libsignal-node-2.0.1.tgz"`（若 pack 生成的文件名不同，改成实际路径）
  3. 执行 `pnpm install --no-frozen-lockfile`

## 3. 避免 `gateway:dev` 自动触发失败的重建

若你已有可用的 `dist/entry.js`（例如之前成功 build 过），希望直接启动、不要自动重建，可保留或写入 buildstamp，让脚本认为 dist 未过期：

```bash
# 若 dist/entry.js 已存在，可写入/更新 stamp 减少自动重建（builtAt 为毫秒时间戳）
echo '{"builtAt":'$(($(date +%s)*1000))',"head":""}' > dist/.buildstamp
pnpm gateway:dev
```

## 4. 控制台日志时间为本地时区

`start.sh` 会从系统读取当前时区（如 macOS 的 `/etc/localtime`）并**仅**传入 Gateway 进程（`env TZ=... pnpm gateway:dev`），不修改系统或当前 shell 的 TZ，日志即按本地时间输出。

若你**未用 start.sh** 而手动执行 `pnpm gateway:dev`，且日志时间为 UTC，可仅对该进程传入 TZ，例如：`TZ=Asia/Shanghai pnpm gateway:dev`（时区用 IANA 名称）。无需改系统配置。
